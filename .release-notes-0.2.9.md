## 0.2.9

Patch release on top of 0.2.8. Three user-reported fixes; no schema migration.

### Fixed

- **"Remove Edit with Elementor from admin bar" actually works now.** Elementor 4.0.x registers the `elementor_edit_page` node late in the `admin_bar_menu` lifecycle (after Site Editor children are appended via filter), so the previous priority-999 hook fired *before* the node existed and removed nothing. Switched the primary hook to `wp_before_admin_bar_render`, which fires after every `admin_bar_menu` callback has run. Added a CSS belt-and-braces hide (`#wp-admin-bar-elementor_edit_page{display:none!important}`) emitted via `wp_head`/`admin_head` so the item stays hidden even if some third party re-adds the node after our removal.
- **No more duplicate Elementor telemetry toggles.** The three component features (`f.elementor.telemetry`, `f.elementor.api_fetcher`, `f.elementor.ai_module`) used to appear as their own cards on Features in addition to the Dashboard "Block Elementor telemetry" composite. They now carry a `hidden: true` flag in the registry and are filtered out client-side by Features and the Dashboard quick-wins picker. They remain in the REST response so the composite toggle can still read and write their individual states.
- **Site Scan results show up live instead of after a page reload.** The polling tick now refreshes `aggregate` + `results` alongside `status` every 1.5 seconds, so partial results accumulate progressively during the scan and the final state lands immediately. The previous behaviour only refreshed on the running → complete transition, which could be missed when object caches briefly returned `idle` between the last batch and the final state write.

### Internal

- New `FeatureMetadata::is_hidden()` plus a corresponding `hidden` field on the REST serialization. Clients can filter as they see fit; composite cards keep full read access.
